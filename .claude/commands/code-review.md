# 코드 리뷰

대상: $ARGUMENTS

## 지시사항

아키텍처 관점에서 코드 리뷰를 수행합니다.

### 1. 대상 파일 확인
- `$ARGUMENTS`가 파일 경로면 해당 파일을 읽습니다
- `$ARGUMENTS`가 비어있으면 최근 변경된 파일들을 검토합니다 (`git diff --name-only HEAD~1`)

### 2. 리뷰 관점

#### 아키텍처 & 설계
- [ ] SOLID 원칙 준수 여부
- [ ] 적절한 계층 분리 (Controller/Service/Repository)
- [ ] 의존성 방향 (상위 레이어가 하위 레이어에만 의존)
- [ ] 단일 책임 원칙 위반 여부

#### 코드 품질
- [ ] 네이밍 컨벤션 준수
- [ ] 함수/메서드 길이 적절성 (30줄 이하 권장)
- [ ] 중복 코드 여부
- [ ] 매직 넘버/문자열 사용 여부

#### 보안
- [ ] 민감 정보 노출 (API 키, 비밀번호 등)
- [ ] SQL Injection 취약점
- [ ] XSS 취약점
- [ ] 입력 유효성 검증

#### 에러 처리
- [ ] 적절한 예외 처리
- [ ] 에러 메시지의 적절성
- [ ] 로깅 수준 적절성

#### 성능
- [ ] N+1 쿼리 문제
- [ ] 불필요한 렌더링 (React)
- [ ] 메모리 누수 가능성

### 3. 결과 보고

다음 형식으로 리뷰 결과를 보고합니다:

```markdown
## 코드 리뷰 결과

### 요약
- 검토 파일: [파일 목록]
- 심각도 높음: X개
- 심각도 중간: X개
- 심각도 낮음: X개

### 발견 사항

#### [심각도 높음] 이슈 제목
- 파일: `path/to/file.ts:line`
- 설명: 문제점 설명
- 권장 수정: 구체적인 수정 방안

#### [심각도 중간] 이슈 제목
...

### 긍정적인 부분
- 잘 작성된 부분 언급

### 개선 권장 사항
- 추가적인 개선 제안
```

## 참고
- 프로젝트의 기존 패턴과 컨벤션을 기준으로 리뷰합니다
- 단순한 스타일 문제보다 아키텍처와 로직 문제에 집중합니다
